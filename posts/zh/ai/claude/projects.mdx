## 每次都要重复告诉 Claude 你的背景信息？

你开了一个新对话，打算让 Claude 帮你写一份项目周报。你花 5 分钟写了一段提示词——项目背景、团队结构、本次迭代的目标……写完发现，这和上周写的基本一样。

每个新对话都是白纸一张。没有记忆，没有上下文，没有知识库。你要么反复输入相同的信息，要么得到一个缺乏上下文、不够精准的回答。

**Claude Projects** 改变了这一点。它是一个专属工作空间——你可以把项目背景文档、写作风格指南、自定义指令一次性存进去。在这个项目空间中的每一次对话，Claude 都自动拥有这些上下文。本文从创建项目到高级用法，完整覆盖。

## 准备工作

- [Claude Pro 订阅](https://claude.ai)（$20/月）——Projects 是 Pro 专属功能
- 准备 3-5 份项目相关文档（PDF、Markdown、TXT）
- 一份自定义指令草案（500 字以内，描述项目规则和约束）

## 核心工作流：设置和使用 Projects

### 第 1 步：创建项目空间

在 Claude 左侧栏点击 **Projects** -> **Create Project**。

你需要填写：

| 字段 | 说明 | 示例 |
|------|------|------|
| **项目名称** | 清晰标识项目 | "电商平台重构 - 后端团队" |
| **项目描述** | 出现在项目列表中的摘要 | 1-2 句话说明项目范围 |
| **项目指令** | 核心自定义指令（见第 2 步） | 项目规则、输出偏好、约束条件 |
| **知识库文件** | 上传项目参考文档 | PRD、API 文档、设计稿 |

**创建原则**：项目的粒度建议按"独立工作领域"划分。一个项目对应一个长期持续的工作流，而不是一个临时任务。

### 第 2 步：编写项目指令（Project Instructions）

项目指令相当于你在该项目空间中**永久的 System Prompt**。每条新对话都会自动加载这些指令。指令的质量直接决定了项目空间的价值。

**优秀项目指令示例**（以写作项目为例）：

```
## 项目概况
这是一个面向 CTO 的技术博客项目。所有文章要求技术深度和商业视角兼顾。

## 写作规范
1. 每篇文章必须有核心论点，不只是"介绍某个技术"
2. 开头 3 段必须是吸引人的钩子（问题、数据、矛盾）
3. 技术术语首次出现时给出定义
4. 每 400 字插入一个数据点或案例
5. 结尾必须有 actionable 的建议

## 语言风格
- 正式但不过于学术
- 避免"在我看来""笔者认为"等主观措辞
- 使用短句。平均句长不超过 20 字
- 不使用 emoji

## 输出格式
- 标题层级：## → ### → ####
- 代码块标注语言
- 列表使用有序列表表示优先级顺序
- 重要内容用 **加粗** 但不要滥用

## 知识库
已上传 2 篇历史文章（最佳实践）和 1 份编辑指南。当被问到风格示例时，参考这些文件。
```

**编写原则**：
1. **短则有力**：500-800 字最佳。太短约束不够，太长模型容易忽略部分指令
2. **用肯定句而非否定句**：写"每段开头用加粗句总结"而非"不要写没有加粗的段落"
3. **区分规则和参考**：强制规则（格式、禁止项）放前面，参考信息（风格建议、背景知识）放后面

### 第 3 步：上传知识库文件

在 **Knowledge Base** 区域点击上传。这些文件在项目空间的每条对话中都可用。

**哪些文件值得上传**：

| 文件类型 | 最佳用途 | 注意事项 |
|---------|----------|---------|
| **PRD / 需求文档** | 项目目标、功能范围 | 保持版本最新 |
| **API 文档** | 接口规范、数据模型 | 需要开发辅助时非常有用 |
| **设计稿 / UI 描述** | 前端实现参考 | 可上传截图 |
| **风格指南** | 写作规范、术语表 | 编辑项目必备 |
| **竞品分析** | 市场参考 | 让回答更有行业上下文 |
| **代码片段** | 架构参考、代码规范 | 用于代码审查和生成 |

**知识库大小建议**：上传 3-10 个文件，总计不超过 20MB。文件越多，每次对话消耗的 token 越多（影响使用额度）。

### 第 4 步：在项目空间中开始对话

在项目空间内点击 **Start a new conversation**。现在你的每条消息都在项目指令和知识库的上下文中运行。

**对比测试**——同一问题，有/无项目上下文：

```
无项目空间：
"帮我写一篇关于微服务的文章。"
→ 通用介绍：什么是微服务、优缺点、何时使用。

有项目空间（已配置技术博客项目和写作规范）：
"帮我写一篇关于微服务的文章。"
→ 针对 CTO 读者的深度分析：微服务在 2026 年的实际落地案例、ROI 数据、组织架构适配建议。
```

**为什么第二个回答更好**：不是 Claude 变聪明了，而是项目指令告诉了它"读者是谁""文章要求什么深度""风格怎样"。知识库还提供了参考案例和行业数据。

### 第 5 步：管理多条对话与版本

项目空间内可以开启多条独立对话。每条对话都有自己的上下文，但共享项目指令和知识库。

**最佳实践**：
- 每条对话聚焦一个子任务（如"API 设计讨论""数据库方案对比""周报生成"）
- 给每条对话起有意义的标题方便检索
- 定期回顾项目指令是否需要更新（项目阶段变化时）
- 项目完成后归档（归档后仍可查看，但不再消耗额度）

**停手条件**：当项目指令和知识库稳定后，连续 3 条新对话的输出质量都符合预期且不需要修改指令。

## 参考表：Projects 配置项速查

| 配置项 | 作用 | 最佳实践 |
|--------|------|----------|
| **项目名称** | 标识和检索 | 包含项目名和领域 |
| **项目描述** | 快速辨识 | 1-2 句话 |
| **项目指令** | 永久 System Prompt | 500-800 字，肯定句式 |
| **知识库文件** | 长期参考文档 | 3-10 个文件，&lt;20MB |
| **对话管理** | 子任务隔离 | 每子任务新建对话 |
| **归档** | 暂停项目 | 完工后归档，不解散 |

## 常见问题排查

### Q1: 项目指令在部分对话中被忽略

**原因**：项目指令最多约 2000 字。超出上限的指令会被静默截断。另外，对话过长时指令的权重会降低。
**解决**：确保指令控制在 800 字以内。最重要的规则放在前 300 字。对话中如果感觉 Claude 偏离了指令，说"请重新读取项目指令"。

### Q2: 知识库文件太多，影响回答速度

**原因**：每个知识库文件在对话开始时都会被加载到上下文中。文件越多，初始加载越慢，且消耗更多 token。
**解决**：只保留 3-5 个最核心的文件。不常用的文件在需要时临时上传（作为附件），完成后从项目空间中移除。

### Q3: 同一项目中不同对话的回答不一致

**原因**：虽然共享指令和知识库，但每条对话是独立的，没有跨对话记忆。Claude 不知道另一条对话中讨论了什么。
**解决**：关键决定（如"我们选择了 PostgreSQL"）写在项目指令中或上传为知识库文件。或者手动在关键信息在新对话开始时粘贴进去。

### Q4: 团队协作时，成员使用的项目效果不统一

**原因**：Claude Projects 目前是单人使用，没有实时协作功能。团队成员各自独立开启自己的项目空间。
**解决**：统一导出项目指令和知识库清单，团队每个成员独立创建。定期同步指令更新。

## 专家技巧

- **指令模板化**：为不同类型的项目创建指令模板（写作项目模板、开发项目模板、分析项目模板）。每次新建项目时直接套用模板再微调，节省 80% 时间。
- **知识库版本控制**：在文件名中加入日期（如"PRD-v2-20260501.md"），让 Claude 能区分版本。在指令中写"知识库中的 PRD 是最新版本，以它为准"。
- **用对话反哺指令**：当你在某条对话中多次纠正同一个问题时，说明指令不够精准。把纠正的内容提炼成一条规则，更新到项目指令中。
- **空项目空间的价值**：即使没有任何知识库文件，只设置项目指令（角色和约束）也能大幅提升输出质量。创建"代码审查助手""会议纪要生成器""翻译助手"等轻量项目空间。
- **定期优化**：每两周回顾一次项目指令，删除不再适用的规则，补充新发现的约束。迭代的项目指令比一次性完美指令更有效。

## 总结

你现在掌握了 Claude Projects 的核心用法：

- **创建项目空间**：绑定指令和知识库
- **编写项目指令**：500-800 字，确定角色、规则、格式和约束
- **知识库管理**：上传核心文档，保持精简
- **多对话管理**：子任务隔离，共享上下文

这和使用 ChatGPT Custom GPTs 的思路类似——都是"一次配置、反复使用"。如果你熟悉 Prompt Engineering 中的五要素框架，你会发现项目指令的本质就是一个长期有效的 System Prompt。
