## 还在手动写代码？让 AI 帮你完成

每个开发者都熟悉这个场景：遇到一个 TypeScript 类型错误，你停下手中的工作，打开浏览器搜索 Stack Overflow；重构一个函数需要在 5 个文件之间来回切换；写单元测试时逐行复制粘贴。这些重复劳动每天消耗你 1-2 小时。传统编码模式让你困在「编辑-编译-调试」的慢循环中。

**Cursor** 是一款深度集成 AI 的代码编辑器（基于 VS Code 架构）。它能理解你的整个代码库，在对话中直接修改代码，甚至一次性编辑多个文件。更重要的是，它不只是在补全代码——它在理解你的意图。本教程将带你逐一掌握 Cursor 的核心 AI 功能。

## 准备工作

动手之前，请确保以下条件就绪：

| 项目 | 要求 |
|------|------|
| Cursor 版本 | 0.45 或更高（从 [cursor.com](https://cursor.com) 下载） |
| 账号 | GitHub 或 GitLab 账号（用于登录并激活 AI 功能） |
| 运行环境 | Node.js 18+（用于本教程的示例项目） |
| 项目代码 | 任意本地项目，建议已初始化 Git（方便撤销 AI 的修改） |

**安装步骤**：
1. 访问 [cursor.com/downloads](https://cursor.com/downloads) 下载对应操作系统的安装包
2. 安装完成后启动 Cursor，点击右上角头像图标，选择 Sign in 完成 GitHub 登录
3. 打开命令面板（`Ctrl+Shift+P`），输入 `Cursor: Settings`，确认 AI 功能已启用（AI 面板应显示 "AI Ready" 状态）
4. 克隆或打开一个你熟悉的项目，确保项目文件能被正常索引（底部状态栏显示索引进度）

## 核心操作流程

### 1. AI 对话 —— 理解代码（`Ctrl+L` / `Cmd+L`）

AI 对话是 Cursor 的基础功能，用于对代码进行提问、分析和调试。

```
操作步骤：
① 打开项目，进入任意 .ts 或 .js 文件
② 选中一个函数（例如一个 API 路由处理函数）
③ 按下 Ctrl+L（Mac 为 Cmd+L），右侧弹出对话面板
④ 选中的代码自动作为上下文发送
⑤ 在输入框中输入："解释这个函数做了什么，并指出潜在的问题和优化空间"
```

**为什么选中代码后再问？** 选中代码可以精确限定 AI 的分析范围，避免它根据文件名猜测。如果不选代码，AI 只能看到当前文件的路径和结构，回答质量明显下降。

**预期输出**：AI 返回函数的功能说明、参数含义、返回值类型分析，以及可能存在的错误处理遗漏或类型安全问题。回答末尾通常会附带改进建议的代码片段。

**停止条件**：得到满意的解释后按 `Esc` 关闭面板。如果需要追问（例如 "那如何修复这个类型安全问题"），直接在对话中输入即可——对话历史会保留。

### 2. 内联编辑 —— 直接改代码（`Ctrl+K` / `Cmd+K`）

不需要离开当前编辑位置就能重构、修改或生成代码。

```
操作步骤：
① 选中要修改的代码段（例如一段使用 .then().catch() 的异步函数）
② 按下 Ctrl+K（Mac 为 Cmd+K），光标位置弹出编辑输入框
③ 输入指令："将这段代码改为 async/await 风格，添加 try/catch 错误处理"
④ 等待 AI 生成 diff 预览（通常 2-5 秒）
⑤ 审查改动：绿色 = 新增，红色 = 删除。按 Tab 接受，按 Esc 拒绝
```

**为什么用内联编辑而不是直接手写？** 内联编辑是 Cursor 最高频使用的功能（占日常 AI 操作的 60% 以上）。它直接在光标处生成修改建议，并以 diff 形式展示改动，让你在应用前清楚知道改了哪里。对于重命名变量、转换语法、添加类型注解等机械操作，比手动编辑快 3-5 倍。

**预期输出**：编辑器以绿色显示新增代码、红色显示被删除代码。接受后源文件被修改。

**停止/回退条件**：如果预览不满意，按 `Esc` 取消，修改提示词重试。如果已经按 Tab 应用了不满意的修改，立即按 `Ctrl+Z`（Cmd+Z）撤销。

### 3. Tab 智能补全 —— 预测式编码

Cursor 的补全不是传统的逐词补全——它会根据上下文预测你接下来要写的完整代码块。

```
操作步骤：
① 在一个 .ts 文件中写一个函数签名，如 "async function fetchUserData(userId: string)"
② 在大括号内开始写逻辑，暂停 0.5 秒
③ 灰色补全建议出现在光标右侧
④ 按 Tab 接受整段建议
⑤ 按 Ctrl+→（Mac 为 Cmd+→）只接受下一个单词，手动控制接受粒度
```

**为什么暂停等待？** Cursor 的补全模型需要约 300-500ms 生成建议。持续打字时模型处于等待状态，短暂停顿可以触发模型生成。补全质量与你最近编辑的历史和项目中的类似模式高度相关。

**预期输出**：一段完整的函数体会以灰色文本显示。按 Tab 后写入文件，文本从灰色变为正常颜色。

**停止条件**：不想要建议时继续打字即可忽略。如果已误按 Tab，用 `Ctrl+Z` 撤销。

### 4. AI 一键修复（`Cmd+;` / `Ctrl+Shift+R`）

遇到红色波浪线时，让 AI 直接分析并修复编译/运行时错误。

```
操作步骤：
① 将光标移到报错的代码行（TypeScript 或 ESLint 红色波浪线）
② 在 Windows/Linux 下按 Ctrl+Shift+R，在 Mac 下按 Cmd+;
③ 等待 2-3 秒，AI 自动分析错误并显示修复建议
④ AI 先用文字说明错误原因，然后以 diff 形式展示修复代码
⑤ 按 Tab 应用修复，或按 Esc 取消
```

**为什么这比手动修快？** 传统做法需要阅读报错信息、回忆语法、查文档、手动改正——每次至少 30 秒。Cursor 的修复功能直接从编译器获取错误信息，结合代码上下文在 3 秒内给出精确修复。

**预期输出**：一个包含原因分析和修复代码的信息框。修复通常包括添加缺失的导入、修正类型注解、调整函数签名等。

**停止条件**：如果修复不正确，按 `Esc` 取消，手动修改。如果修复破坏了其他代码，立即 `Ctrl+Z` 回滚。

### 5. Composer 多文件编辑（`Ctrl+Shift+L`）

当需求涉及多个文件时（如添加新功能、重构模块），Composer 可以一次性处理所有文件。

```
操作步骤：
① 按下 Ctrl+Shift+L（Mac 为 Cmd+Shift+L），打开 Composer 面板
② 在输入框中描述需求，例如："在所有 API 路由中添加速率限制中间件，使用 @nestjs/throttler"
③ 如果 AI 没有自动识别相关文件，使用 @file 手动引用："@file src/app.module.ts @file src/cats/cats.module.ts"
④ 按 Ctrl+Enter 或点击 Send 按钮
⑤ AI 列出所有将要修改的文件，每个文件附带 diff 预览
⑥ 逐个文件审查，点击 Accept 接受单个文件，或 Accept All 接受全部
```

**为什么用 Composer 而不是手动改多个文件？** 跨文件重构最容易出错——改了 A 文件忘了改 B 文件、导入路径写错、函数签名不一致。Composer 让你在一个视图中审查所有改动，避免遗漏。

**预期输出**：Composer 面板列出 3-8 个受影响的文件，每个文件显示清晰的 diff。底部显示总体改动统计（新增/删除行数）。

**停止条件**：审查 diff 时发现某文件修改不对，点击该文件旁的 Skip/Reject，不影响其他被接受的修改。

## 常见问题排查

**问题 1：按 `Ctrl+L` 无响应，对话面板打不开**
- **原因**：AI 服务未连接或网络环境受限
- **解决**：检查编辑器右下角状态栏，确认显示 "AI Ready ✓" 字样。若显示 "Offline" 或 "Disconnected"，点击状态栏选择 "Reconnect"。企业网络环境下，确保 WebSocket 连接未被防火墙拦截（需要连接 `*.cursor.com` 和 `*.openai.com`）

**问题 2：Tab 补全始终不出现**
- **原因**：模型文件尚未下载完成，或文件类型未被识别
- **解决**：首次安装后需等待模型下载（约 2-5 分钟，视网速而定）。可以在 Cursor 设置中查看 "Models" 下载进度。确认当前文件的扩展名（.ts、.js、.py、.go 等）在 Cursor 支持的语言列表中。关闭并重新打开文件可以触发索引重建

**问题 3：AI 回答不准确，频繁提及不存在的文件或函数**
- **原因**：没有使用 @ 引用符号限制上下文，AI 猜测了错误的文件；或者代码库索引已过期
- **解决**：始终在提示词中使用 @file 或 @func 精确指定范围。运行命令面板（`Ctrl+Shift+P`）→ "Cursor: Rescan Index" 强制重建索引。检查 `.cursorignore` 是否误将关键目录排除

**问题 4：Composer 修改了不该改的文件**
- **原因**：选择了 "Accept All" 而未逐一审查 diff
- **解决**：立即使用 `git diff` 查看所有改动，用 `git checkout -- <file>` 恢复特定文件。最佳做法：在 Composer 中始终使用单个 Accept 而不是 Accept All，每个文件确认后再提交

**问题 5：内联编辑生成的代码格式混乱，不符合项目规范**
- **原因**：缺少格式化配置，或提示词不够精确
- **解决**：确保项目已配置 ESLint + Prettier。在提示词后追加 "请遵循 ESLint 和 Prettier 配置" 可显著改善输出。如需持久化行为，在项目根目录创建 `.cursorrules` 文件

## 专家建议

**时间节省数据**：根据实际开发测量，Cursor 的内联编辑可将常见重构任务（重命名、提取方法、转换语法）从 2-3 分钟缩短到 20-30 秒。Tab 补全每天约可减少 200-400 次按键，相当于节省约 30 分钟/天。

**最佳实践**：
- **@ 引用是核心技巧**：在对话中始终使用 `@file`、`@func`、`@sym` 引用具体代码元素。实测表明，精确引用可将回答准确率从 40% 提升至 85% 以上
- **项目级 `.cursorrules`**：在项目根目录创建 `.cursorrules` 文件，声明技术栈（React + TypeScript + Tailwind）、代码风格（函数式组件、命名约定）和关键库。AI 回复会更贴合项目实际
- **先选中再操作**：无论是 Ctrl+L 对话还是 Ctrl+K 编辑，都先选中相关代码段。选中后 AI 的上下文感知准确度远高于纯文字描述
- **复杂任务拆成步骤**：与其一次输入 "构建一个完整的用户管理系统"，不如分解为 "1. 创建 User 实体 2. 添加 CRUD 路由 3. 编写单元测试"。分步执行的准确度更高
- **修改前先提交**：在让 AI 执行大规模多文件修改前，先用 `git commit` 保存当前进度。AI 改错后的回滚成本几乎为零

**常见陷阱**：
- **网络环境**：Cursor 的 AI 功能需要联网，代码片段会发送到云端模型处理。敏感项目请确认公司政策允许
- **检查导入语句**：AI 生成的 import/require 语句经常出错（路径错误、命名导出混淆），应用后务必检查
- **长对话退化**：单次对话超过 20 轮后，AI 的上下文利用效率会下降。遇到复杂的后续问题，建议开启新对话并重新引用相关文件
