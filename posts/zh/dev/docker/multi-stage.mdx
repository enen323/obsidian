> **版本**: Docker Engine 17.05+（多阶段构建自 17.05 引入）
> **难度**: ★★★☆☆（需掌握 Dockerfile 基本语法和镜像概念）
> **用时**: 25-35 分钟

## 300MB 的镜像里，只有 10MB 是你写的代码

一个典型的 Node.js 生产镜像：`FROM node:18`（350MB），装完依赖（200MB），复制源码（2MB）。最终镜像大小约 550MB，其中你的业务代码只占不到 0.5%。

同样的问题出现在 Go、Java、Python 应用中——构建时需要的编译工具链、头文件、开发依赖在生产运行时完全不需要，却被一起打包进了最终镜像。

**多阶段构建（Multi-stage Build）** 是 Docker 17.05+ 引入的功能，允许在一个 Dockerfile 中使用多个 `FROM` 语句，每个 `FROM` 开始一个新阶段。你可以在第一个阶段安装所有构建工具、编译代码，然后在第二个阶段只复制编译产物到精简的基础镜像中。最终镜像只包含运行所需的最小文件集。

根据 Docker 官方案例，多阶段构建通常能减少 60-90% 的镜像体积。本教程将通过三个语言的实战案例展示具体操作。

## 准备工作

- **Docker Engine** 17.05+（推荐 24+）
- 基本的 Dockerfile 编写能力
- 熟悉 `docker build` 和 `docker images` 命令
- 以下每个语言案例可以独立练习，选择一个你熟悉的即可

## 实战案例

### 案例一：Node.js 应用（最常用）

#### 单阶段构建（对比组）

```dockerfile
# Dockerfile.single
FROM node:18

WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

EXPOSE 3000
CMD ["node", "dist/index.js"]
```

构建并查看大小：

```bash
docker build -f Dockerfile.single -t node-app:single .
docker images node-app
# 大小约 1.2GB（node:18 全量版 + devDependencies + build 产物）
```

#### 多阶段构建

```dockerfile
# Dockerfile
# ====== 阶段 1：构建阶段 ======
FROM node:18 AS builder

WORKDIR /app
COPY package*.json ./
# 安装所有依赖（包括 devDependencies）
RUN npm install
COPY . .
# 构建（TypeScript 编译、打包等）
RUN npm run build

# ====== 阶段 2：运行阶段 ======
FROM node:18-alpine

WORKDIR /app
# 只从 builder 阶段复制构建产物和 production 依赖
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/package*.json ./
# 现在只安装生产依赖
RUN npm install --production

EXPOSE 3000
CMD ["node", "dist/index.js"]
```

```bash
docker build -t node-app:multi .
docker images node-app
# 大小约 150MB（node:18-alpine 约 120MB + production 依赖约 30MB）
# 相比单阶段节约约 85%
```

**关键行**：
- `FROM node:18 AS builder`：给阶段命名，后续用 `--from=builder` 引用
- `COPY --from=builder /app/dist ./dist`：只复制构建产物，不复制 `node_modules` 的 200MB 开发依赖
- 运行阶段使用 `node:18-alpine`（约 120MB），而非 `node:18`（约 350MB）

#### 进一步优化：依赖安装分层

上述 Dockerfile 还有一个问题：每次 `npm install` 重新安装所有依赖。利用 Docker 的层缓存进一步优化：

```dockerfile
FROM node:18 AS builder
WORKDIR /app

# 第一步：只复制依赖描述文件并安装依赖
# （依赖不常变，这一层会被缓存）
COPY package*.json ./
RUN npm install

# 第二步：复制源码并构建
# （源码常变，但不会触发上层缓存失效）
COPY . .
RUN npm run build

FROM node:18-alpine
WORKDIR /app

COPY --from=builder /app/dist ./dist
COPY --from=builder /app/package*.json ./
RUN npm install --production

EXPOSE 3000
CMD ["node", "dist/index.js"]
```

**为什么这样写**：只要 `package.json` 不变化，`RUN npm install` 就使用缓存层，构建时间从 2 分钟降到 5 秒。

### 案例二：Go 应用（极致精简）

Go 编译为静态二进制文件，完全不依赖运行时，理论上可以做到极小镜像。

#### 多阶段构建

```dockerfile
# ====== 阶段 1：编译 ======
FROM golang:1.22-alpine AS builder

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
# CGO_ENABLED=0 禁用 CGO，确保静态编译
RUN CGO_ENABLED=0 GOOS=linux go build -o /app/server ./cmd/server

# ====== 阶段 2：运行 ======
# 使用 scratch（完全空的镜像），只包含二进制文件
FROM scratch

# 拷贝静态文件（如有）
COPY --from=builder /app/server /server
# 拷贝时区数据（如有需要）
COPY --from=builder /usr/share/zoneinfo /usr/share/zoneinfo

EXPOSE 8080
CMD ["/server"]
```

```bash
docker build -t go-app:multi .
docker images go-app
# 大小约 15MB（Go 二进制文件通常 10-20MB）
# 相比 golang:1.22-alpine（约 300MB+）节约 95%
```

**为什么用 `scratch`**：scratch 是一个空镜像，没有任何操作系统层、没有 shell、没有包管理器。对于 Go 这样的静态编译语言，这是最安全、最精简的基础镜像——攻击面最小。

**备选 `distroless` 基础镜像**：如果应用需要 SSL 证书、ca-certificates 或 /etc/passwd，可以用 Google 的 distroless 镜像代替 scratch：

```dockerfile
FROM gcr.io/distroless/static-debian11:nonroot
COPY --from=builder /app/server /server
CMD ["/server"]
```

### 案例三：Python 应用

```dockerfile
# ====== 阶段 1：构建 ======
FROM python:3.12-slim AS builder

WORKDIR /app
COPY requirements.txt ./
# 构建 wheels 到 /wheels 目录
RUN pip install --user --no-cache-dir -r requirements.txt

# ====== 阶段 2：运行 ======
FROM python:3.12-slim

WORKDIR /app
# 只复制 Python 库，跳过构建时缓存的 pip 数据
COPY --from=builder /root/.local /root/.local
COPY . .

# 确保 PATH 包含用户安装的 Python 包
ENV PATH=/root/.local/bin:$PATH

EXPOSE 8000
CMD ["gunicorn", "app:app"]
```

**Python 优化的关键**：将 pip 安装的依赖从 `--user` 安装目录复制过来，而非从 `site-packages` 复制。避免将 pip 本身的缓存、setuptools 等构建工具带到最终镜像。

**对比**：单阶段 `python:3.12-slim` + pip install => 约 350MB；多阶段 => 约 180MB（节约约 50%）。

## 基础镜像对比表

| 基础镜像 | 大小 | 内容 | 适用语言 |
|----------|------|------|----------|
| `scratch` | 0B | 完全空 | Go 静态编译 |
| `alpine` | ~5MB | Alpine Linux + musl + apk | 通用（需注意 musl vs glibc） |
| `distroless` | ~15MB | 静态链接库 + SSL 证书 | Go、Java、Python |
| `debian:stable-slim` | ~80MB | Debian + glibc + apt | 通用（兼容性最好） |
| `ubuntu:22.04` | ~100MB | Ubuntu + glibc + apt | 通用 |
| `node:18-alpine` | ~120MB | Node.js + Alpine | Node.js |
| `node:18` | ~350MB | Node.js + Debian | Node.js（兼容性优先） |

## 多阶段构建高级技巧

### 1. 命名阶段构建缓存

```dockerfile
FROM node:18-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --production

FROM node:18-alpine AS build
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build

FROM node:18-alpine
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY --from=build /app/dist ./dist
CMD ["node", "dist/index.js"]
```

将依赖安装拆为独立阶段，多个后续阶段可共用。

### 2. 按目标平台条件构建（BuildKit 特性）

```dockerfile
# syntax=docker/dockerfile:1.4
FROM node:18-alpine AS base
WORKDIR /app

FROM base AS dev
RUN npm install
COPY . .
CMD ["npm", "run", "dev"]

FROM base AS prod
COPY package*.json ./
RUN npm ci --production
COPY . .
RUN npm run build
CMD ["node", "dist/index.js"]
```

构建时选择目标：

```bash
# 开发镜像（包含所有开发依赖）
docker build --target dev -t my-app:dev .

# 生产镜像（只含生产依赖）
docker build --target prod -t my-app:prod .
```

### 3. COPY --from 的高级用法

```dockerfile
# 从外部镜像复制文件
COPY --from=nginx:alpine /etc/nginx/nginx.conf ./nginx.conf

# 从宿主机上的其他构建结果复制
COPY --from=/path/to/binary /app/server /server
```

## 常见问题

### 问题 1：多阶段构建后镜像大小没有明显减少

**症状**：辛苦写了多阶段 Dockerfile，`docker images` 显示镜像大小仍然是 500MB+。

**原因**：
- 可能原因一：运行阶段仍使用全量基础镜像（如 `FROM node:18` 而非 `node:18-alpine`）
- 可能原因二：`COPY --from=builder` 复制了不必要的文件（如整个 `node_modules`）
- 可能原因三：Docker 保留了中间构建缓存，用 `docker images` 看到的可能是旧镜像

**排查和解决**：

```bash
# 1. 用 docker image history 查看每层大小
docker image history my-app:multi

# 2. 用 dive 工具可视化分析镜像层（推荐）
docker run --rm -it -v /var/run/docker.sock:/var/run/docker.sock wagoodman/dive my-app:multi

# 3. 确保使用 alpine 版本
FROM node:18-alpine  # 而非 node:18

# 4. 检查 COPY 路径是否正确
COPY --from=builder /app/dist ./dist  # 而非 COPY --from=builder /app .
```

### 问题 2：Alpine 镜像中运行 Node.js 原生模块报错

**症状**：`node:18-alpine` 中运行需要本地编译的模块（如 `sharp`、`bcrypt`）时，出现 `Error: module initialization failed` 或 `Cannot find module`.

**原因**：Alpine 使用 musl libc 而非 glibc，某些原生 C++ 扩展没有预编译的 musl 版本，需要额外安装编译工具链。

**解决**：

```dockerfile
FROM node:18-alpine
# 安装编译原生模块所需的系统依赖
RUN apk add --no-cache python3 make g++

# 或者：切换到 slim 版本（基于 Debian，使用 glibc）
FROM node:18-slim
```

如果实在需要兼容性，使用 `node:18-slim`（约 200MB）而非 `node:18-alpine`。

### 问题 3：COPY --from 找不到文件

**症状**：构建时报错 `COPY --from=builder failed: file not found in build context or stage`。

**原因**：
- 前一阶段的指定路径不存在
- 前一阶段中文件路径写错（比如构建到 `/app/server`，但复制时写成了 `/server`）
- 前一阶段的命令执行失败但未报错（如 `RUN npm run build` 输出到了不同的目录）

**解决**：

```dockerfile
# 在构建阶段添加验证步骤
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY . .
RUN go build -o /app/server ./cmd/server && ls -la /app/server  # 添加 ls 验证

FROM scratch
COPY --from=builder /app/server /server
CMD ["/server"]
```

### 问题 4：BuildKit 缓存失效导致每次都完整构建

**症状**：即使没有修改 package.json，`RUN npm install` 每次都重新执行。

**原因**：`docker build` 默认使用传统构建器，对某些场景的缓存判断不准确。或 `COPY` 的源文件有元数据变化（如 git 操作改变了时间戳）。

**解决**：

```bash
# 启用 BuildKit
DOCKER_BUILDKIT=1 docker build -t my-app .

# 或设置默认启用
echo 'export DOCKER_BUILDKIT=1' >> ~/.bashrc
```

在 Dockerfile 开头添加注释以启用 BuildKit 语法：

```dockerfile
# syntax=docker/dockerfile:1.4
FROM node:18-alpine AS builder
```

## 专家建议

- **使用 dive 分析镜像**：`dive` 工具可以可视化展示每一层增加了哪些文件，是优化镜像体积的必备武器。安装：`brew install dive` 或 `docker run wagoodman/dive`。
- **层数 vs 层大小**：优化重点是"大层"而非"层数"。减少一个 200MB 的层比合并两个 1KB 的层有价值得多。Docker 的层数限制为 127 层，但大多数应用远达不到这个限制。
- **依赖安装策略**：node 项目用 `npm ci`（比 `npm install` 快 2-3 倍且可重现）；Python 用 `pip install --no-cache-dir`；Go 利用 `go mod download` 的缓存。核心原则：将依赖安装放在源码复制之前。
- **安全考量**：scratch 和 distroless 镜像由于没有 shell，攻击者即使利用应用漏洞也无法获得交互式 shell。这是"纵深防御"的最佳实践。CI/CD 流水线中加入 `docker scout` 扫描：

```bash
docker scout quickview my-app:prod
```

- **CI/CD 集成建议**：在多阶段构建的 CI 流水线中，利用 Docker 的 BuildKit 内置缓存挂载（`--mount=type=cache`）加速构建：

```dockerfile
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN --mount=type=cache,target=/root/.npm npm ci
COPY . .
RUN npm run build
```

## 总结

本教程通过 Node.js、Go、Python 三个实战案例展示了多阶段构建的核心模式：构建阶段（完整工具链 + 所有依赖）→ 运行阶段（精简基础镜像 + 仅复制产物）。同时介绍了命名阶段、BuildKit 条件构建、dive 分析等高级技巧。

案例实践：
- Node.js：`FROM node:18 AS builder` + `FROM node:18-alpine`，节省 ~85%
- Go：`FROM golang:alpine AS builder` + `FROM scratch`，节省 ~95%
- Python：`FROM python:slim AS builder` + `FROM python:slim`，节省 ~50%
